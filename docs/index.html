Secure Print

End-to-end krypterad säker utskrift med smartkortautentisering
Windows / Linux IPP Everywhere  →  CUPS + S3 + PostgreSQL  →  YubiKey  →  Skrivare
Teknisk systemöversikt · 2026

Problemet – traditionell nätverksutskrift

Säkerhetsbrister

  • Utskriftsjobb skickas i klartext på nätverket
  • Jobben köar i klartext på skrivarens hårddisk
  • Vem som helst kan plocka upp ett utskrivet dokument
  • Inga krav på autentisering vid hämtning
  • Ingen spårbarhet – vem skrev ut vad och när?

Krav på lösning

  • Dokumentet krypterat hela vägen – aldrig i klartext på nätverket
  • Bara rätt person kan dekryptera (privat nyckel på smartkort)
  • Fysisk närvaro krävs – användaren måste vara vid skrivaren
  • Fullständig auditlogg för compliance
  • Driverless – fungerar direkt från Windows och Linux utan extra installation

Komponenter

Klientdator

  • Windows 10/11 – IPP Everywhere inbyggt, ingen driver
  • Linux – CUPS / lpr / KDE/GNOME-dialoger, IPP-stöd inbyggt
  • Skrivare konfigureras via ipps://cups-server/printers/s3-queue
  • TLS-krypterad transport till CUPS

CUPS

  • Tar emot IPP-jobb från klienter via IPPS (443)
  • Kör custom backend: s3print
  • Autentiserar klienter via Kerberos / SPNEGO
  • Stateless – all state i PostgreSQL + S3
  • Kan köras som Deployment på Kubernetes

s3print-backend

  • Python-script som CUPS anropar
  • Hämtar cert från LDAP/AD
  • Krypterar med openssl cms
  • Laddar upp till S3
  • Registrerar metadata i PostgreSQL

Lagring

  • S3 – krypterade jobfiler (.cms)
  • PostgreSQL – jobbmetadata, status, UPN
  • Jobb raderas automatiskt efter 48 h

Utskriftsterminal

  • Linux mini-PC + pekskärm
  • PyQt6 – native app, ingen browser
  • YubiKey via PKCS#11 (opensc)
  • Kopplar direkt till S3 och PostgreSQL
  • Inga lyssnande portar

Skrivare

  • IPP Everywhere-kompatibel (PWG)
  • Terminal skriver ut via lpr lokalt
  • Alla IPP-alternativ bevaras

Krypteringsflöde – end-to-end

Vid utskrift (klientdator)

1. Klienten (Windows/Linux) skickar PDF/PS
   via IPP/TLS till CUPS s3-queue

2. s3print-backend:
   a. Hämtar användarens X.509-cert
      från Active Directory (LDAP)
   b. openssl cms -encrypt \
        -aes-256-gcm \
        -binary \
        -recip user.pem
   c. Laddar upp .cms-fil till S3
   d. Skriver jobbpost i PostgreSQL
      status = 'pending'
Hybridkryptering AES-256-GCM (AEAD) RSA för sessionsnyckeln

Vid hämtning (terminal + YubiKey)

3. Användaren sätter i YubiKey + PIN

4. python-pkcs11: autentiserar mot
   kortets privata nyckel,
   extraherar UPN från SAN-fältet

5. Hämtar .cms-fil från S3

6. openssl cms -decrypt \
     -binary \
     -provider pkcs11
   → privata nyckeln lämnar aldrig kortet

7. Skriver ut dekrypterad fil med lpr

8. Markerar jobbet 'retrieved' i DB
Privat nyckel på hårdvara PIN-skyddad OCSP/CRL-kontroll

Autentisering – Kerberos & tjänstekonton

Klient → CUPS via Kerberos

1. Klienten har ett Kerberos TGT från AD
   (Windows: automatiskt · Linux: kinit)

2. Klienten begär ett tjänstebiljett för
   HTTP/cups-server@COMPANY.COM (SPN)

3. IPP-anropet bär SPNEGO/Negotiate-header
   med den krypterade Kerberos-biljetten

4. CUPS validerar biljetten mot sin keytab
   → extraherar autentiserat principal
   → user@COMPANY.COM

5. Principalnamnet skickas som argv[2]
   (username) till s3print-backend

6. s3print använder UPN för att slå upp
   userCertificate i Active Directory
Ingen lösenordsinmatning på klienten SSO med befintlig AD-inloggning

IPP Everywhere – drivrutinsfritt

Hur det fungerar

  • Standardiserat av PWG (Printer Working Group) – RFC 8011
  • Windows / Linux skickar PDF direkt – inga konverteringssteg
  • CUPS s3-queue använder custom backend – ingen PPD, ingen filterkjedja
  • Alla IPP-alternativ lagras verbatim i databasen och spelas upp vid utskrift

Inga drivrutiner behövs

  • Inte på klientdatorn (Windows eller Linux)
  • Inte på terminalen
  • Terminalen skriver ut direkt via lokal lpr

Kapabiliteter – ett viktigt designval

⚠ Klienten pratar med s3-queue, inte med den fysiska skrivaren

get-printer-attributes returnerar s3-queue:ns generiska kapabiliteter — inte den fysiska skrivarens. Klientens utskriftsdialog visar alltså ett generiskt alternativset (duplex, A4, färg/mono m.m.) snarare än skrivarspecifika alternativ (specifika pappersfack, häftpositioner osv.).

Hur det hanteras: s3-queue konfigureras i CUPS med breda generiska attribut. Alla IPP-alternativ användaren väljer lagras verbatim som JSON i PostgreSQL. Vid utskrift på terminalen spelas de upp som lpr -o key=value mot den lokala skrivaren, som avgör om den stöder varje alternativ.
Bevarade alternativ (exempel)
media · sides · print-color-mode · print-quality · number-up · finishings · page-ranges · orientation-requested
Skrivar­agnostisk kö Samma kö – flera terminaler Generisk print-dialog

Kubernetes – stateless CUPS-driftsättning

Varför stateless fungerar

  • CUPS-containern håller inget eget jobbstate – allt lagras i PostgreSQL och S3
  • Varje s3print-anrop är atomärt: läs från LDAP → kryptera → skriv S3 → skriv DB
  • Flera CUPS-pods kan köra parallellt utan delad disk eller delat minne
  • En pods krasch innebär max ett tappt jobb (CUPS-retries hanterar det)
Vad som inte är stateless: PostgreSQL och S3 är externa tjänster med eget HA-upplägg (RDS Multi-AZ, S3-replikering m.m.) — de hanteras utanför Kubernetes-klustret.

Kubernetes-resurser

apiVersion: apps/v1
kind: Deployment
metadata:
  name: cups-s3print
spec:
  replicas: 3          # horisontell skalning
  template:
    spec:
      containers:
      - name: cups
        image: secure-print/cups:latest
        envFrom:
        - secretRef:
            name: s3print-secrets  # LDAP, S3, DB-creds
        ports:
        - containerPort: 631       # IPPS via Ingress → 443
Horisontell skalning Rolling updates Secrets via K8s Secret Keytab via K8s Secret volume Ingress → TLS-terminering

Nyckeldesignbeslut

  • Privat nyckel lämnar aldrig YubiKey (PKCS#11 on-card crypto)
  • Native PyQt6 – ingen webbserver, ingen browser-attackyta
  • Stateless CUPS – körbart på Kubernetes, horisontellt skalbart
  • Stateless terminal – all state i S3 + PostgreSQL
  • Kerberos SSO – klienten autentiseras med befintlig AD-inloggning
  • IPP Everywhere – noll klientinstallation på Windows och Linux
